Questions

After completing the lab recap as a class the following questions:

General Discussion Questions

Q: Where does a service broker have to be be deployed?
A: The broker can be deployed provided the Cloud Controller can reach it via HTTPS.

Q: Why provide unique credentials per binding?
A: So that unbinding (revoking access) doesn’t affect multiple applications.

Questions From the Lab

Q: Can a service broker support upgrade/downgrade of a service instance to a different plan?
A: Upgrade is supported.

Q: The broker is required by the Cloud Controller to respond within how many seconds?
A: 60 seconds.

Q: Does provisioning have to be done synchronously?
A: No, it can be done asynchronously.

Q: Do all services have to be bindable?
A: No. Need an example here.

Troubleshooting

Troubleshooting guidance for the lab that instructors need to know.

Name and Id Collisions

Make sure that students set the environment variables to unique values for the entire platform, otherwise they won’t be able to create/register their service broker.

Purge When Necessary

If a student’s service broker/service instance/bindings get completely messed up and normal methods no longer work (create-service, bind-service, unbind-service, delete-service) there are options to essentially reset things.

For instance, if they can’t delete/unbind a service instance consider the following methods (does not require admin for Spaced-Scoped Brokers):

// removes one service instance and associated bindings
$ cf purge-service-instance

//removes the service from the marketplace and all service instances with associated bindings
$ cf purge-service-offering
